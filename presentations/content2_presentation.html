Part 1: Understanding Subsearches

What is a Subsearch?

A subsearch is a search within a search that:

  • Runs first and passes its results to the outer (main) search
  • Enclosed in square brackets [ ]
  • Returns values that act as filters or inputs for the main search
  • Has performance implications - limited to 50,000 results by default
Key Concept: Subsearches enable dynamic filtering - the outer search uses results from the inner search as criteria

Subsearch Syntax & Structure

Basic Syntax

index=outer_index [search index=inner_index | fields field_name]

Execution Flow

  1. Inner search runs first: [search index=inner_index | fields field_name]
  2. Results passed to outer search: Becomes field_name=value1 OR field_name=value2...
  3. Outer search executes: Using subsearch results as filter
Important Limits:
  • Max results: 50,000 (configurable in limits.conf)
  • Max time: 60 seconds (configurable)
  • Nested subsearches: Maximum 3 levels deep

When to Use Subsearches

Perfect Use Cases

  • Dynamic Filtering: Filter main search based on results from another search
  • Correlation: Find related events across different indexes or time ranges
  • Top-N Analysis: Find events related to top performing entities
  • Anomaly Detection: Compare against baseline or threshold values

Example: Find Web Requests from IPs with Failed Logins

index=web status=200 [search index=auth action=failure | stats count by src_ip | where count > 5 | fields src_ip] | stats count by uri

This finds successful web requests from IPs that had more than 5 failed login attempts

Subsearch Commands & Techniques

1. The return Command

Controls how many results and which fields the subsearch returns

index=main [search index=users premium=true | return 1000 $user_id]
  • return N - Returns N results (default 50,000)
  • return $field - Returns specific field value
  • return N $field - Returns N values of specific field

2. The format Command

Customizes how subsearch results are formatted for the outer search

index=web [search index=users status=active | fields user_id | format "(" "user=" "\" OR \"" ")" ]

Creates: (user="user1" OR user="user2" OR user="user3")

Advanced Subsearch Patterns

Pattern 1: Time-Based Correlation

index=web [search index=app error earliest=-1h | stats earliest(_time) as start_time | return $start_time] | stats count by status

Finds web logs starting from when the first app error occurred

Pattern 2: NOT with Subsearches

index=web NOT [search index=whitelist | fields src_ip] | stats count by src_ip

Finds web traffic from IPs NOT in whitelist

Pattern 3: Multiple Field Matching

index=transactions [search index=fraud detected=true | eval search_string=user_id."::".transaction_id | fields search_string]

Subsearch Performance Optimization

Best Practices

  1. Minimize Results: Use stats, dedup, or head in subsearch
  2. Specific Fields: Always use | fields to limit returned fields
  3. Time Range: Limit subsearch time range when possible
  4. Consider Alternatives: Sometimes join or stats are more efficient
Optimization Example:
BAD: [search index=app | fields user_id] GOOD: [search index=app earliest=-1h | dedup user_id | fields user_id | head 10000]
Warning: Subsearches that return too many results will be automatically truncated, potentially missing relevant data

Part 2: Search Macros

What are Search Macros?

Search macros are reusable search snippets that:

  • Store commonly used search logic
  • Accept arguments for flexibility
  • Improve search consistency and maintainability
  • Can be shared across teams

Benefits

  • Reusability: Write once, use everywhere
  • Consistency: Standardize search patterns across organization
  • Maintainability: Update macro once, affects all uses
  • Abstraction: Hide complexity behind simple names

Creating Search Macros

Method 1: Via Web UI

  1. Settings → Advanced Search → Search macros
  2. Click "New Search Macro"
  3. Enter name, definition, and optional arguments
  4. Set permissions (Private/App/Global)

Method 2: Via macros.conf

[my_macro_name] definition = index=web status>=400 | stats count by uri iseval = 0
Naming Convention: Use descriptive names like failed_web_requests or high_cpu_servers

Using Search Macros

Basic Macro (No Arguments)

Definition:

Name: failed_web_requests Definition: index=web status>=400

Usage:

`failed_web_requests` | stats count by uri

Note the backticks (`) around the macro name

Parameterized Macro (With Arguments)

Definition:

Name: status_filter(1) Arguments: status_code Definition: index=web status=$status_code$

Usage:

`status_filter(404)` | stats count by src_ip

Advanced Macro Patterns

Multi-Argument Macros

Name: date_range_filter(2) Arguments: start_date, end_date Definition: earliest=$start_date$ latest=$end_date$

Usage:

index=web `date_range_filter("2024-01-01","2024-01-31")` | stats count

Nested Macros

Macro 1: production_servers Definition: host IN (prod-web-01, prod-web-02, prod-app-01) Macro 2: prod_errors Definition: `production_servers` error OR failed

Usage:

index=app `prod_errors` | stats count by host

Macro Best Practices

  • Naming: Use clear, descriptive names (e.g., high_priority_alerts)
  • Documentation: Add descriptions in the macro definition
  • Scope: Start with private, expand to app/global when stable
  • Validation: Validate arguments to prevent injection issues
  • Testing: Test macros thoroughly before promoting to production
  • Versioning: Consider version numbers for major changes (e.g., alert_v2)
Security Note: Be cautious with macros that accept user input - validate and sanitize arguments

Part 3: Statistical Commands

Overview

Statistical commands transform raw events into meaningful metrics:

  • stats - General-purpose statistical aggregation
  • chart - Creates tabular data for visualization
  • timechart - Time-series statistical analysis
  • eventstats - Adds stats as fields to original events
  • streamstats - Cumulative statistics
Key Difference: transforming commands (stats, chart, timechart) replace events with statistics, while enriching commands (eventstats, streamstats) add stats to existing events

The stats Command

Basic Syntax

... | stats () as by

Common Statistical Functions

  • count: Count number of events
  • sum: Sum of numeric values
  • avg: Average of values
  • min/max: Minimum/maximum values
  • stdev: Standard deviation
  • values: List of unique values
  • list: List of all values (including duplicates)
  • dc (distinct_count): Count of unique values

Examples

index=web | stats count by status index=web | stats avg(response_time) as avg_response by uri index=web | stats sum(bytes) as total_bytes, avg(bytes) as avg_bytes by host

Advanced stats Techniques

Multiple Aggregations

index=web | stats count as requests, avg(response_time) as avg_response, max(response_time) as max_response, sum(bytes) as total_bytes by host, status

Percentiles

index=web | stats perc50(response_time) as median, perc95(response_time) as p95, perc99(response_time) as p99 by uri

Using eval with stats

index=web | stats count as total, count(eval(status>=400)) as errors, count(eval(response_time>1000)) as slow_requests by host

The chart Command

Purpose

Creates data suitable for charts and tables with over/by clauses

Syntax

... | chart over by

Examples

index=web | chart count over status by host index=web | chart avg(response_time) over uri by host index=sales | chart sum(revenue) over product by region

chart vs stats

  • stats: General purpose, simple grouping
  • chart: Two-dimensional pivoting (rows and columns)
  • chart: Better for creating visualizations

The timechart Command

Purpose

Statistical analysis over time - creates time-series data

Basic Syntax

... | timechart [span=] () by

Time Span Options

  • span=1m - 1 minute buckets
  • span=5m - 5 minute buckets
  • span=1h - 1 hour buckets
  • span=1d - 1 day buckets
  • Auto-determined if not specified

Examples

index=web | timechart count index=web | timechart span=5m avg(response_time) index=web | timechart count by status

Advanced Statistical Analysis

eventstats - Enriching Events

Adds statistical calculations as fields without removing events

index=web | eventstats avg(response_time) as avg_response by host | eval performance = if(response_time > avg_response*2, "slow", "normal")

streamstats - Running Calculations

Calculates cumulative or windowed statistics

index=sales | streamstats sum(revenue) as running_total index=web | streamstats window=10 avg(response_time) as moving_avg

Use Cases

  • eventstats: Comparing individual values to aggregates
  • streamstats: Trend analysis, moving averages

Statistical Functions Reference

Function Description Example
count Number of events stats count by host
sum Total of values stats sum(bytes) by uri
avg Average value stats avg(response_time)
min/max Minimum/maximum stats min(price) max(price)
stdev Standard deviation stats stdev(response_time)
values List unique values stats values(user) by host
dc Count distinct stats dc(user) as unique_users
earliest/latest First/last value stats latest(_time) by user

Real-World Statistical Analysis Examples

Example 1: Web Performance Dashboard

index=web | stats count as requests, avg(response_time) as avg_response, perc95(response_time) as p95, count(eval(status>=500)) as server_errors, dc(src_ip) as unique_visitors by uri | sort -requests

Example 2: Hourly Transaction Volume

index=transactions | timechart span=1h sum(amount) as revenue, count as transaction_count, avg(amount) as avg_transaction

Example 3: Anomaly Detection

index=app | timechart span=5m count | eventstats avg(count) as avg_count, stdev(count) as stdev_count | eval threshold=avg_count+(stdev_count*3) | eval anomaly=if(count>threshold, "yes", "no")

Common Statistical Patterns

Pattern 1: Top N Analysis

index=web | stats count by uri | sort -count | head 10

Pattern 2: Percentage Calculations

index=web | stats count as total, count(eval(status>=400)) as errors | eval error_rate=(errors/total)*100

Pattern 3: Comparison to Average

index=web | stats avg(response_time) as response by host | eventstats avg(response) as overall_avg | eval performance=case( response < overall_avg*0.8, "excellent", response < overall_avg*1.2, "normal", 1==1, "poor")

Pattern 4: Rate of Change

index=metrics | timechart span=1h sum(value) as total | delta total as change | eval change_pct=(change/total)*100

Performance Considerations

Optimize Statistical Queries

  • Filter Early: Use index, sourcetype, host filters before stats
  • Limit Fields: Only include necessary fields in calculations
  • Use prestats: For distributed searches, use prestats=true
  • Appropriate Spans: Don't use overly granular time spans
  • Limit Results: Use head or tail after stats when appropriate
Best Practice:
BETTER: index=web status=200 | stats count by uri WORSE: index=web | stats count by uri, status | where status=200
